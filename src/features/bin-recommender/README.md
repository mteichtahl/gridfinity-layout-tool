# bin-recommender

Static lookup table that predicts a bin's `WxDxH` size from its user-entered label, trained from production ML telemetry. Powers the inline "ghost bin" suggestion in the drawer.

## Contract

```ts
import { recommendBinSize } from '@/features/bin-recommender';
import model from '@/features/bin-recommender/model.json';

const prediction = recommendBinSize({
  label: 'screws',
  drawer: { width: 8, depth: 12, height: 4 },
  model,
});
// prediction: { size: { width, depth, height }, p: 0.62, n: 41, source: 'label' } | null
```

The consumer is responsible for loading `model.json` — when this hooks into the UI (next PR), it will be dynamically imported so the chunk is only fetched when the Labs flag is on.

The function returns `null` whenever:

- The model's `schemaVersion` doesn't match the runtime (forward-incompat artifact)
- The model's `vocabVersion` doesn't match `VOCAB_VERSION` from the telemetry module (model was trained against a different vocab — short-circuits one-shot with a warning)
- The label is blank or whitespace AND there's no drawer-prior with enough support
- Every source in the resolution cascade has fewer than its minimum-sample threshold

## Resolution cascade

| Tier       | Source                  | Threshold | Used when                                                                                                      |
| ---------- | ----------------------- | --------- | -------------------------------------------------------------------------------------------------------------- |
| PRIMARY    | `byLabelHash[hash]`     | `n ≥ 10`  | Label hashes match exactly. Handles any language because the hash is computed on `label.toLowerCase().trim()`. |
| ENRICHMENT | `byEmbedBucket[bucket]` | `n ≥ 20`  | OOV labels. Same 4-char semantic bucket the telemetry already records — covers synonyms and minor typos.       |
| FALLBACK   | `byDrawer["WxDxH"]`     | `n ≥ 50`  | Nothing else hits. Drawer-size prior — what people put in drawers of this size, regardless of label.           |

`pickTop()` returns the top-ranked size only (the JSON keeps top-3 for future tie-breaking but the runtime currently uses just the top entry).

## Training

The model artifact (`model.json`) is regenerated by `scripts/train-bin-recommender/train.py`. See that directory's README for the workflow. The initial committed artifact is empty — `recommendBinSize` returns `null` on every call until the first real training run produces data.

## Why client-side?

- **Zero API cost.** The recommender ran ~250k inferences during local QA testing; doing that against a serverless endpoint would crush the Redis command budget.
- **Offline-safe.** PWA users (the app's primary deployment surface) keep working without a network.
- **Privacy-clean.** No label text leaves the device for the suggestion path — the hash is computed client-side and matched against pre-computed buckets.

## Bundle impact

The consumer is expected to dynamically `import('./model.json')` from inside the Labs-gated code path so Vite code-splits it into its own chunk. Users without the `bin_size_recommender` Labs flag never load it.
